## Что сейчас не так

1. **Публикации зашиты в код**: 4 поста живут в `src/lib/site.ts` как массив `PUBLICATIONS`. Чтобы добавить новый — нужно править исходники.
2. **В админке раздел отключён**: в `src/routes/admin.tsx` пункт «Публикации» помечен `enabled: false` («скоро»).
3. **Страница `/blog` пустая**: сейчас там стоит заглушка «Раздел наполняется» с кнопкой Telegram. Самого списка публикаций на `/blog` нет (хотя они показываются на главной и открываются по прямой ссылке `/blog/{slug}`).
4. **Даты в карточках на главной нет**: в `HomePublications.tsx` карточка стилизована «как Telegram-пост» и показывает только заголовок + текст без даты публикации.

То есть та же проблема, что была с прогнозами: публикации и админка — две разные сущности.

## Что будет сделано

### 1. База данных: таблица `publications`

Создаём таблицу в Lovable Cloud — единый источник правды для всех мест, где показываются публикации.

Поля:
- `id` (uuid)
- `slug` (text, unique) — для URL `/blog/{slug}`
- `title` (text) — заголовок
- `tag` (text) — рубрика (Мнение / Разбор / ИИ и работа / Прогнозы)
- `excerpt` (text) — короткое описание
- `body` (text) — полный текст поста
- `published_at` (date) — **дата публикации** (это то, чего вам не хватает в карточках)
- `telegram_url` (text, nullable) — ссылка на оригинал в Telegram
- `is_visible` (boolean, default true) — глазок «показать / скрыть» как у прогнозов
- `display_order` (int) — для ручной сортировки если понадобится
- `created_at`, `updated_at`

RLS-политики (по образцу `predictions`):
- читать может кто угодно (public read);
- создавать / редактировать / удалять — только `admin` (через существующую `has_role`).

Миграция данных: автоматически перенесём 4 текущие публикации из `site.ts` в новую таблицу при первой миграции (одним `INSERT`-ом), чтобы вы их сразу увидели и в админке, и на сайте — ничего не пропадёт.

### 2. Личный кабинет: новый раздел `/admin/publications`

Включаем пункт «Публикации» в сайдбаре админки (сейчас `enabled: false` → станет `true`).

Создаётся страница `src/routes/admin.publications.tsx` с тем же UX, что и у прогнозов:

- **список всех публикаций** в виде карточек с заголовком, датой, тегом и превью;
- кнопка **«Новая публикация»** → форма (заголовок, slug — авто-генерация из заголовка, тег, дата публикации через DatePicker, короткое описание, полный текст в `<textarea>`, ссылка на Telegram);
- **редактирование** существующих (карандаш на карточке);
- **удаление** (корзина) с подтверждением;
- **глазок видимости** (Eye / EyeOff) — как у прогнозов: скрытая публикация исчезает с публичной части, но остаётся в админке;
- **realtime**: как только сохраняете изменение — публичная `/blog` и главная страница обновляются без перезагрузки.

### 3. Публичная страница `/blog`

Заменяем текущую заглушку на полноценный список:

- хедер «Публикации» оставляем;
- ниже — лента всех видимых (`is_visible = true`) публикаций, отсортированных по `published_at` (новые сверху);
- каждая карточка: дата + тег + заголовок + короткое описание + стрелка → ведёт на `/blog/{slug}`;
- по визуалу — переиспользуем существующий компонент `PublicationsList` (который уже верстает строки с датой и тегом), просто кормим его данными из БД, а не из `site.ts`, и снимаем лимит «4 штуки».

### 4. Главная страница: дата в карточке поста

В `HomePublications.tsx` (карточки в стиле Telegram) добавляем **дату публикации** под именем канала — там, где сейчас стоит только `@neuromein`. Будет: `@neuromein · 20 ноября 2025`. Это то, чего вам не хватает в карточках.

Сами карточки на главной тоже начнут читать из БД (а не из `site.ts`), чтобы новые посты, добавленные через ЛК, появлялись и здесь, и на `/blog`, и при открытии `/blog/{slug}`.

### 5. Страница отдельного поста `/blog/{slug}`

Перевод с `PUBLICATIONS.find(...)` на запрос к БД. URL не меняется, дизайн не меняется — меняется только источник данных. Полный текст статьи (`body`) тоже отображается, а не только excerpt-плейсхолдер как сейчас.

### 6. Навигация

Пункт «Публикации» в верхнем меню сайта **уже есть** (см. `NAV_LINKS` в `src/lib/site.ts`) и ведёт на `/blog` — отдельно ничего добавлять не нужно. Просто сама страница `/blog` теперь будет содержательной.

## Технические детали

**Файлы — создаются:**
- `supabase/migrations/<timestamp>_publications.sql` — таблица + RLS + сид существующих 4 постов + включение `supabase_realtime`.
- `src/data/publications.fetch.ts` — функция `fetchPublications(opts?: { onlyVisible?: boolean })`.
- `src/routes/admin.publications.tsx` — CMS-страница (CRUD + visibility toggle + DatePicker для `published_at`).

**Файлы — редактируются:**
- `src/routes/admin.tsx` — `enabled: true` для пункта «Публикации».
- `src/routes/blog.tsx` — заглушка → реальный список из БД через `PublicationsList`.
- `src/routes/blog.$slug.tsx` — loader читает из БД через `fetchPublicationBySlug(slug)`, отображает `body`.
- `src/components/PublicationsList.tsx` — принимает `items` пропсом (вместо импорта `PUBLICATIONS`); убираем `.slice(0, 4)`.
- `src/components/HomePublications.tsx` — данные из БД, в карточке-«Telegram-посте» добавляется `dateLabel` рядом с `@neuromein`.
- `src/lib/site.ts` — массив `PUBLICATIONS` остаётся как тип-схема и резервная копия, но из компонентов больше не импортируется (после миграции данных в БД).

Realtime: подписка на канал `publications-public` в `/blog`, `/` и `/blog/{slug}` на события `INSERT/UPDATE/DELETE` — мгновенный re-fetch.

## Результат

- В личном кабинете появляется рабочий раздел **«Публикации»**: добавить, отредактировать, скрыть/показать, удалить.
- Дата публикации — обязательное поле формы, она показывается **и в карточках на главной, и в списке на `/blog`, и в шапке отдельного поста**.
- Любое изменение в ЛК мгновенно отражается на `/blog`, на главной и на странице самой публикации.
- 4 текущих поста никуда не пропадают — сразу видны и в админке, и на сайте.
- В дальнейшем по этой же схеме подключим оставшиеся разделы (исследования, FAQ).
